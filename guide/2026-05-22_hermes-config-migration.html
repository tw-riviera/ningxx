Hermes Config 遷移指南

從舊版 config.yaml 到新結構：一次完整的對照與遷移手冊。

01為什麼需要遷移

Hermes Agent 的設定檔結構在版本迭代中經歷了顯著變化。舊版 config.yaml 採用扁平化結構，將所有 auxiliary provider 直接列在根層級；新版則引入了 providers 區塊、明確的 key_env 對應、以及更細緻的 auxiliary 子模組配置。

遷移不是「可有可無」——舊結構在新版啟動時可能導致 auxiliary service 無法正確解析，或 fallback 機制失效。

核心差異

舊版：auxiliary 直接指定 provider 名稱，無獨立 provider 定義區塊。
新版：先定義 provider（含 base_url、key_env、api_mode），再在 auxiliary 中引用 provider 名稱。

03新版結構（遷移後）

新版結構將 provider 定義抽離，並在 auxiliary 中改為引用已定義的 provider 名稱：

model:
  default: kimi-k2.6
  provider: kimi-primary-coding
  base_url: https://api.kimi.com/coding

providers:
  kimi-primary-coding:
    base_url: https://api.kimi.com/coding
    key_env: KIMI_CODING_API_KEY
    api_mode: anthropic_messages
    model: kimi-k2.6
  kimi-moonshot:
    base_url: https://api.moonshot.ai/v1
    key_env: KIMI_API_KEY

04對照表：舊 → 新

項目 舊版 新版
主模型 provider kimi-coding kimi-primary-coding（自定義名稱）
provider 定義 無獨立區塊 providers: 區塊，含 key_env、api_mode
auxiliary provider kimi-coding kimi-moonshot（區分主模型與輔助服務）
fallback 隱含或未啟用 明確 fallback_providers 區塊
auxiliary 欄位 provider / model / base_url 新增 api_key、timeout、extra_body、max_concurrency 等
環境變數 隱式對應 顯式 key_env 指定

05遷移步驟

  1. 備份現有 config
    cp /opt/data/config.yaml /opt/data/config.yaml.backup.$(date +%Y%m%d)
  2. 確認環境變數
    檢查 .env 中是否已有 KIMI_CODING_API_KEY 與 KIMI_API_KEY。
  3. 建立新版 config
    參考上方「新版結構」範本，依實際環境調整 provider 名稱與 base_url。
  4. 驗證 auxiliary 服務
    啟動 Hermes 後，觀察 vision / session_search / compression 是否正常運作。
  5. 測試 fallback
    暫時停用主 provider，確認系統能否自動切換至 fallback provider。
注意

新版 auxiliary 中每個子項都包含 api_key: ''——留空表示從 key_env 指定的環境變數讀取。若直接填入明文金鑰，將覆蓋環境變數設定。

06常見問題

Q: 為什麼 auxiliary provider 要改用 kimi-moonshot？

A: 主模型（kimi-primary-coding）與輔助服務（vision、compression 等）可以分屬不同 endpoint。將 auxiliary 指向 moonshot 可分散負載，也便於個別調整 timeout 與並發限制。

Q: api_mode: anthropic_messages 是什麼意思？

A: 指定該 provider 使用 Anthropic Messages API 格式進行通訊，而非 OpenAI-compatible 格式。Kimi Coding endpoint 目前採用此格式。

Q: 遷移後舊檔案要保留嗎？

A: 建議保留。系統目前仍可能讀取 /opt/data/config.yaml 作為輔助參考，但啟動時以新版結構為準。
