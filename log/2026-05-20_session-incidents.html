🔴 事件一：Context Compression 404 錯誤 (2026-05-20)

錯誤症狀

CRITICAL
用戶端看到：
Moonshot 404: kimi-k2-turbo-preview...
在对话过程中突然出现，导致上下文压缩完全失败。
後果：對話無法自動摘要，session 越來越長，最終可能觸發 token limit。

🔍 調查過程

1. 讀取 /opt/data/config.yaml，發現 auxiliary.compression.provider 設為 auto

2. 深入原始碼 agent/auxiliary_client.py，追蹤 auto 解析邏輯

3. 發現關鍵 hardcoded fallback：

_API_KEY_PROVIDER_AUX_MODELS_FALLBACK = {
    "kimi-coding": "kimi-k2-turbo-preview",
    # ... 其他 provider
}

4. 確認 kimi-k2-turbo-preview 是無效/不存在的模型名，API 回傳 404

🛠 解決方案

在 config.yaml 明確指定 auxiliary 任務的 provider 與模型，禁止依賴 auto： 
auxiliary:
  vision:
    provider: moonshot
    model: kimi-k2.6
  compression:
    provider: moonshot
    model: kimi-k2.6
  # ... 其他 auxiliary 任務同樣設置

重啟指令：/restart 或 hermes gateway restart（config 在啟動時 snapshot，熱更新無效）

🧠 Hermes 理解長上下文的五大策略

當對話愈來愈長，Hermes 不會無限制塞滿 context window。它透過以下五層機制，在「記住一切」與「不超限」之間取得平衡。

1. Current Context

當前對話的即時記錄，包含最近數輪用戶與 AI 的完整對話。這是最精確、最即時的記憶層。

2. Compaction Summary

當對話過長時，系統調用輔助模型將歷史對話壓縮為摘要，騰出空間繼續對話。

3. Persistent Memory

用戶主動保存到 memory 的關鍵資訊，跨 session 永久保留。分 user 記憶與 memory 筆記。

4. Session Search

當前 session 缺少上下文時，自動搜尋過往 sessions 的摘要，將相關內容注入當前對話。

5. Archive Files

所有歷史對話的原始記錄（.jsonl），供深度檢索或調試使用，是最完整的備份層。

各策略的工作方式與局限性

Current Context（當前上下文）

運作
每條訊息即時追加到當前 session 的 context window 中，包含完整對話內容與 tool 調用結果。
局限
受模型 token limit 限制（kimi-k2.6 約 64K~256K）。當對話過長，最早內容會被遺忘或需壓縮。

Compaction Summary（上下文壓縮摘要）

觸發條件
compression.threshold（預設 0.5）達標時，或手動執行 /compress 時觸發。
運作流程
1. 調用 auxiliary model 讀取整個對話歷史
2. 生成濃縮摘要，保留關鍵決策、程式碼、結論
3. 用摘要替換原始對話，騰出 token 空間
我們的教訓
當 auxiliary.compression.provider: auto 解析到無效模型 kimi-k2-turbo-preview 時，整個壓縮機制癱瘓，session 持續膨脹直至無法回應。務必明確指定 auxiliary provider。

Persistent Memory（持久化記憶）

運作
每次對話開頭，系統讀取 /opt/data/memory.json，將 user profile 與 memory notes 注入 system prompt，確保個人偏好跨 session 保留。
局限
容量有限（約 ~2,000 chars），只適合存「用戶偏好、程式碼風格、專案設定」等精簡事實，不能存對話歷史或任務進度。
使用方式
調用 memory(action="add") 或 memory(action="replace")。永遠優先儲存「會多次影響未來互動」的資訊。

Session Search（會話搜索）

運作
當用戶提及「上次」、「之前」或系統偵測到語義缺口時，呼叫 session_search(query="..."，從所有歷史 sessions 中檢索相關對話摘要。
局限
摘要品質取決於 compaction 是否成功。如果壓縮失敗（如本事件），session search 的回溯能力大受影響。此外，它是被動觸發而非主動關聯。
我們的教訓
Daily session reset 後（UTC 12:00 HKT），如果 compaction 在前一晚失敗，次日 session search 可能找不到關鍵上下文，導致用戶感到「斷線」。

Archive Files（歸檔文件）

運作
每個 session 完整記錄在 /opt/data/sessions/YYYYMMDD_HHMMSS_xxxx.jsonl 中，包含時間戳、工具調用、模型回應等原始數據。
局限
純存儲，不主動參與推理。需要用戶或系統主動調用 session_search 或讀取檔案才能利用。
清理建議
保留 request_dump_*.json（網路請求調試）可安全刪除以釋放空間，但 *.jsonl 是對話歷史，刪除後無法回溯。

Compaction 何時觸發？

觸發條件說明結果
自動觸發 當 context_length / max_length > threshold（預設 0.5） 無縫壓縮，用戶無感知
手動觸發 用戶輸入 /compress 立即壓縮，可用於主動控制
Session 重啟 Daily reset 或 /restart 後 新 session 讀取舊 session 的 summary
壓縮失敗 auxiliary provider 出現 404 或其他錯誤 Context 持續膨脹，最終可能超限

Auto-Lineage：壓縮後的 Session 延續

當一個 session 被壓縮後，Hermes 創建一個「繼承 session」，標題自動附加 #2、#3：

原始 session: "Void Mathematics 設計討論"
    │
    ▼ 壓縮後
新 session: "Void Mathematics 設計討論 #2"
    │
    ▼ 再壓縮
新 session: "Void Mathematics 設計討論 #3"

每次延續都帶上前一個 session 的 summary，實現長期連續工作流。

關鍵教訓

2026-05-20
Auto 模式不可靠
provider: auto 的 hardcoded fallback 可能指向無效模型。使用自架/非主流 provider 時務必明確指定。
2026-05-20
Config 需重啟生效
修改 config.yaml 後必須 /restart，因為 auxiliary client 在啟動時 snapshot 配置，不接受熱更新。
持續
監控所有 auxiliary 任務
除了 compression 和 vision，還有 session_search、web_extract、title_generation 等。建議一次性全部明確配置。
長期
理解 Context 層級的取捨
· Current Context：最準確但 ephemeral
· Compaction Summary：持久但可能丟失細節
· Persistent Memory：精準但容量小
· Session Search：回溯強但依賴壓縮品質
· Archive：最完整但不主動參與推理

快速診斷 Checklist

⚠️ 如果再次遇到「對話突然斷線」或「AI 好像忘了之前說的」：

  1. 檢查 /opt/data/config.yaml 的 auxiliary provider 是否為 auto → 改為明確指定
  2. 重啟 gateway：hermes gateway restart
  3. 檢查 /opt/data/sessions/ 是否有異常大的 .jsonl 檔案（表示壓縮失敗）
  4. 檢查 archive 目錄確保歷史對話有正常存檔
  5. 定期清理 request_dump_*.json 釋放空間
