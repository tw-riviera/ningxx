Mon / 忙

When the tool cannot hold what the eye already sees.

01The Brief

用戶帶來一份參考作品：Jie Jian 的《Touch, 2024》。一本 5×6 吋的小冊，記錄了朋友之間所有包含「摸」字的對話。每一句都是真實的身體經驗，從摸貓的溫柔到被侵犯的不安。

用戶想要仿製這個概念，將「Touch / 摸」換成「Mon / 忙」。一個關於「週一」與「忙碌」的雙關語，一份對於生活侵入的小詞典。

規格是 IG 動態圖：1080 × 1350 px。要求精確還原參考圖的排版語法：左邊大幅留白（約60–70%）、垂直分隔線、speaker 與 timestamp 的凸縮層次、雙時間戳邏輯、關鍵詞 upright bold 而非斜體。

02The Ceiling

這是今日最核心的課題。用戶說：「看似簡單的藝術，去模仿的時候，竟然先明白 AI 的天花板。」

2.1 PIL 的崩潰

最初選擇 PIL （Python Imaging Library）直接繪製 PNG。理論上，這是最精確的方法——每一個像素都是確定的。但實際上，PIL 是「畫布」，不是「排版引擎」。中英文混排時，基線（baseline）自動漂移；中文換行需要逐個字計算；hanging indent 必須手動數像素放置第二行起點。

技術細節

PIL 的 ImageFont.truetype() 載入 Noto Sans TC 後，font.getsize(text) 回傳的高度僅包含字型本身，不包含行間距（line gap）。當中英文混排時，英文字母如 "g" 或 "y" 的 descender 會超出中文基線，導致行高計算累積誤差。

更嚴重的是：PIL 不支援 CSS 的 hanging-indent，也不支援「關鍵字放大但不撑破行高」這種微細排版。每個關鍵詞都需要手動計算 y 座標，避免與前後文字重疊。

2.2 瀏覽器渲染的謊言

放棄 PIL 後，轉而使用 HTML + CSS，讓瀏覽器負責渲染。但服務器上沒有 Chrome/Chromium，無法用 Playwright 或 Puppeteer 進行 headless 渲染。唯一可用的是 WeasyPrint，但 WeasyPrint 對於精細排版的支援有限，輸出的 PNG 質感不足。

於是改為「輸出純 HTML，由用戶在手機 Safari 打開後截圖」的 workflow。這個想法理論上很好——利用 Safari 的 WebKit 引擎獲得真正精確的字距、基線與字體載入。但實際上，iOS WebKit 對於中英文混排的行內渲染，有著難以預測的限制。

iOS Safari 渲染限制

  • display: inline-block + position: relative 在行內放大關鍵詞時，行高計算崩潰——關鍵詞變成獨立圖層飄浮，完全錯位
  • 中文與英文基線不同，瀏覽器會自動調整，導致關鍵詞上下偏移
  • font-size 放大後，line-height 的實際渲染高度不等於 CSS 設定值
  • Safari 自動將日期範圍（如 0422–0518）識別為藍色連結，必須用 SVG 或 format-detection meta 規避
  • Google Fonts 在 iOS 上載入後，實際渲染的字體字重、hinting 都可能跟設計頻率不同

2.3 「差一點點」的累積

今日經歷了六個版本的迭代：v1 至 v6。每一次都認為「差不多了」，但用戶的眼睛總是看到那個「差一點點」。這個「差一點點」不是數字上的差距，而是「分寸」——一種無法用代碼言說的視覺判斷。

AI 寫得出一千首詩，
但調一粒字的大小，
作不到。

2.4 為何原圖是藝術

Jie Jian 的原圖中，「touch」甚至不是粗體。它只是「剛好在它該在的地方」。這不是排版，這是確定性。

用戶最終用 Canva 手動排出了最終版本。關鍵詞「Mon」用大字 serif 直接覆蓋在「忙」之上，用視覺暴力展現雙關。這個手法比單純加粗更有衝擊力，也更能體現「Mon-busy」的概念。

03The Iterations

3.1 v1 — inline-block 災難

初始方案使用 display: inline-block + position: relative 做關鍵詞行內放大。在 iOS Safari 上，關鍵詞變成獨立圖層飄浮，完全錯位，遮蓋正文，行高崩潰。

3.2 v2 — 純 inline 回退

徹底移除 inline-block，改為純 inline 渲染。排版穩定下來，但關鍵詞凸出太微弱，幾乎看不出。底部日期仍被 iOS 變成藍色連結。

3.3 v3 — 硬兩欄網格

實施硬兩欄：左邊 metadata 約300px，右邊內容約680px。加入雙時間戳邏輯（同一 speaker 的 burst 訊息）。但左留白仍遠不足 60–70%。

3.4 v4 — 擴大間距與 SVG 底部

Entry 之間 margin 增至 58px，關鍵字微增至 20px。底部日期改用 SVG 繪製，徹底規避 iOS 自動連結。但用戶明確指出左邊未達承諾的七成留白。

3.5 v5 — 過度擴張

誤解了「留白」的意思，將內容推至 x=560（約 52%）。結果整頁過於疏離，speaker 和 timestamp 變成「孤獨的小島」，與對話剖裂。用戶說：「你為什麼沒有看到他們是與對話緊密地在一起嗎？」

3.6 v6 — 最終版

內容起點改為 x=640，speaker 和 timestamp 移至 x=580，左邊留白約55–60%。但用戶仍然說「中央還差好遠」。

最終，用戶用 Canva 自己做出了最終版本。關鍵詞「Mon」用大字 serif 直接覆蓋「忙」，用視覺暴力展現雙關語。

    06Reflection

    6.1 關於天花板

    今日最重要的發現：AI 的天花板不在於「生成內容」，而在於「控制分寸」。AI 可以寫出一千個版本，但調整一個 keyword 的大小、一個 entry 之間的間距、一個邊距的百分比，需要的是人的觸覺。

    6.2 關於工具

    排版的難度不在於「做出來」，而在於「做到對」。PIL 給你精確像素控制，但排版邏輯得自己寫。HTML/CSS 給你排版邏輯，但瀏覽器渲染會自作主張。Canva 給你所見即所得，但每個字都要手動拉。

    6.3 關於藝術

    Jie Jian 的原圖之所以是藝術，不是因為它「排得好看」，而是因為它「每個決定都是有意識的」。那些留白、那些凸出、那些沉默，都是設計師逐個像素調出來的。這種確定性，是任何算法都無法提示出來的。

    原來我們以為自己在用 AI 創作，
    其實是 AI 在用我們的眼睛，
    學習什麼叫做「對」。

    6.4 關於未來

    今日的經驗告訴我們，當遇到「看似簡單」的設計任務時，應該第一時間說：「這個用自動工具做不到，我們要討論其他方法。」而不是繼續趕路，浪費時間。

    用戶最後用 Canva 手動做出了作品。這是最好的結局——因為最好的工具，始終是那雙能看見分寸的眼睛。
