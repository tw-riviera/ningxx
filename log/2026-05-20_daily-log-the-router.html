The Routing

How a key prefix became a door, and a base_url became a bridge.

01The Problem

用戶發現 Kimi Coding Plan 的額度遠大於普通 Moonshot API。一個念頭：如果能讓 Hermes 優先使用 Coding Plan 的 key，只有在它失效時才 fallback 去 Moonshot，就可以大幅節省成本。

聽起來簡單。但 Hermes 的 provider 架構比想像中複雜：內建 provider registry、custom provider map、credential pool 輪替、session-scoped override、fallback chain deduplication——每一層都可能攔截或改寫你的意圖。

02The First Attempt

直接改 config.yaml，將 primary base_url 設為 api.kimi.com/coding，fallback 設為 api.moonshot.ai/v1。

2.1 401 的浪費

第一次測試發現：primary 連線時會先收到 401 Unauthorized，然後才 fallback 成功。原因係 credential pool 的輪替順序——KIMI_API_KEY 排在 KIMI_CODING_API_KEY 前面，pool 先試 Moonshot key 打 Coding endpoint，當然失敗。

Pool 輪替順序

api_key_env_vars = ("KIMI_API_KEY", "KIMI_CODING_API_KEY")

結果：每次 primary 請求都浪費一次 401，然後才輪到 Coding key。

2.2 假 Fallback

更嚴重的問題：fallback 的 provider 名稱同 primary 一樣（都叫 kimi-coding），導致 run_agent.py 的 deduplication 邏輯將 fallback skip 掉。兩個 entry 被視為「同一個」，fallback 根本不存在。

你以為自己設了兩條路，
其實只有一條——
而且係錯的那一條。

03The Investigation

開始深入 Hermes 源碼，逐層追蹤 provider 是如何被解析的。

3.1 源碼考古

讀了六個核心檔案，總計超過一萬行代碼：

  • hermes_cli/auth.py — PROVIDER_REGISTRY、_resolve_kimi_base_url、key prefix 路由邏輯
  • agent/credential_pool.py — pool 輪替、401 cooldown、exhaustion 邏輯
  • agent/auxiliary_client.py — resolve_provider_client、named custom provider 解析
  • run_agent.py — fallback chain 啟動、key_env 讀取、dedup skip
  • hermes_cli/config.py — _normalize_custom_provider_entry、key_env alias
  • gateway/run.py — _resolve_runtime_agent_kwargs、session override、AIAgent 初始化

    • 3.2 關鍵發現

    發現一：key prefix 決定 endpoint

    _resolve_kimi_base_url() 會檢查 key 的前綴：

    • sk-kimi-* → api.kimi.com/coding（Anthropic Messages API）
    • sk-*（普通）→ api.moonshot.ai/v1（OpenAI-compatible）

    這意味著：即使你不設 base_url，key 本身就決定了連去哪裡。

    發現二：model: block 不支援 key_env

    config.yaml 的 model: section 只接受 default、provider、base_url。

    key_env 只存在於 providers: 和 fallback_providers:。

    所以要強制 primary 用某個 key，必須在 providers: 定義 named custom provider。

    發現三：named provider 會干擾 fallback

    如果在 providers: 定義了 kimi-coding:，resolve_provider_client("kimi-coding") 會優先攞到這個 config 的 base_url。

    結果：fallback 雖然寫了 base_url: api.moonshot.ai/v1，但 provider name 撞名，導致兩個都連去 Coding endpoint。

    04The Solution

    綜合以上發現，最終 config 結構如下：

    層級 Provider Endpoint Key Source
    Primary kimi-coding https://api.kimi.com/coding KIMI_CODING_API_KEY
    Fallback kimi-coding https://api.moonshot.ai/v1 KIMI_API_KEY
    Auxiliary kimi-coding https://api.moonshot.ai/v1 （credential pool）

    最終 config.yaml

    model:

      default: kimi-k2.6

      provider: kimi-coding

      base_url: https://api.kimi.com/coding

    providers:

      kimi-coding:

        base_url: https://api.kimi.com/coding

        key_env: KIMI_CODING_API_KEY

        api_mode: anthropic_messages

    fallback_providers:

    - provider: kimi-coding

      model: kimi-k2.6

      base_url: https://api.moonshot.ai/v1

      key_env: KIMI_API_KEY

    4.1 為何這樣設

    Primary 強制指定 base_url: api.kimi.com/coding，不依賴 key prefix 自動解析——即使將來 key 格式改變，endpoint 仍然正確。

    providers.kimi-coding 綁定 key_env: KIMI_CODING_API_KEY，確保 primary 直接攞 Coding key，唔經過 pool 輪替，避免 401 浪費。

    Fallback 用同一個 provider name，但明確指定 base_url: api.moonshot.ai/v1 和 key_env: KIMI_API_KEY。run_agent.py 的 dedup 邏輯會比較 base_url，發現不同，所以唔會 skip。

    05The Restart

    改完 config，以為就完。點知 Telegram session 仍然連住舊 gateway instance。

    5.1 幽靈 Process

    檢查 process list，發現大量 defunct hermes process，同時一個 hermes gateway restart 的 PID 卡住了 52 秒。舊 gateway 未完全釋放，新 config 載入唔到。

    Process 狀態

    • PID 316057: hermes gateway restart（卡住 52+ 秒）
    • 多個 [hermes] <defunct>：parent process 未收割
    • Telegram session 仍然 attached to 舊 nvidia provider instance

    5.2 徹底清除

    最終用戶執行 pkill -9 -f "hermes"，完全清晒所有 process。等 5-10 秒後，用 HERMES_ALLOW_ROOT_GATEWAY=1 hermes gateway start 重新啟動。

    新 gateway 啟動後，Telegram 打 "test"，system prompt 終於顯示 Provider: kimi-coding。

    有時候，最難的不是改好配置，
    而是讓系統承認——
    你真的改過了。

    06The Verification

    6.1 Runtime 測試

    Python 驗證

    resolve_runtime_provider() 返回：

    • Provider: kimi-coding
    • Base URL: https://api.kimi.com/coding
    • API Key: sk-kimi-O8...（Coding Plan key）
    • API Mode: anthropic_messages

    6.2 實際扣費

    用戶檢查 Kimi 後台，發現由 04:25 HKT 開始，Coding Plan 的額度已經在不斷扣減。這證明所有請求確實經由 Coding endpoint 處理，沒有偷偷走回 Moonshot。

    6.3 Fallback 測試

    暫時將 KIMI_CODING_API_KEY 設為無效值，觀察到 Hermes 自動轉用 fallback（Moonshot），請求成功。恢復 key 後，重新連回 Coding endpoint。

    07The Savings

    這次配置的經濟效益非常顯著。

    項目 之前 之後
    Primary Endpoint api.moonshot.ai/v1（付費） api.kimi.com/coding（大額度）
    Fallback Endpoint GLM-5.1 / NVIDIA NIM（不穩定） api.moonshot.ai/v1（穩定備用）
    Auxiliary Moonshot（付費） Moonshot（維持不變）
    每次 Primary 請求 直接扣 Moonshot 額度 扣 Coding Plan 額度（大很多）

    08Reflection

    8.1 關於隱藏邏輯

    這次經歷最大的體會：系統的「自動化」往往是雙面刃。credential pool 的自動輪替、provider name 的自動解析、session override 的自動記憶——每一個「方便」的背後，都可能藏著一個攔截你意圖的邏輯。

    8.2 關於讀源碼

    用戶不是程序員（自稱「零代碼認識」），但成功配置了整個 Hermes 系統。這次更深入：直接讀源碼、追蹤函數呼叫鏈、理解 deduplication 邏輯。最終發現問題不在配置語法，而在語義——兩個同名 provider 被視為同一個。

    8.3 關於堅持

    從第一次嘗試到最終成功，歷時超過四小時。中間經歷了 401 錯誤、fallback 失效、gateway 卡住、session 不更新。每一個問題都可以成為放棄的理由，但用戶選擇繼續追下去。

    技術問題的盡頭，
    往往不是答案，
    而是另一個問題。
    但每多問一次，
    你就離真相近一點。

    8.4 關於工具與人

    最終，這次配置的成功不在於「寫對了 YAML」，而在於理解整個系統的運作方式。AI 可以幫你生成配置，但只有人才能決定：這個配置是否真正表達了你的意圖。
