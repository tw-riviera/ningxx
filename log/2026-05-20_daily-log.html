03The Knowledge

3.1 什麼是 Context Compression？

LLM 每次對話都有 token 上限（context window）。當對話越來越長，接近上限時， Hermes 會自動將舊訊息「壓縮」成一段摘要（summary），騰出空間給新對話。 這個 summarization 不是本地做，而是 call 一個「輔助模型」（auxiliary model）來幫手寫摘要。

技術細節

Compression 流程：

  • 系統偵測 context 接近 threshold（預設 90%）
  • 抽出舊訊息，交給 auxiliary.compression 指定的模型
  • 模型回傳一段 summary，替換掉原有的一堆訊息
  • 如果 compression 失敗，系統會 insert 一個 fallback marker， 但舊訊息仍佔用 token，只是「隱藏」了

失敗的後果：summary 寫唔到 → 舊訊息冇被壓縮 → context 繼續膨脹 → 最終被迫開新 session 或截斷內容。

3.2 Auxiliary Model 是什麼？

Hermes 將主模型（你傾緊嘅 model）和輔助模型分開。 輔助模型負責「幕後雜務」：寫摘要、生成標題、分析圖片、搜尋歷史對話、審批命令... 它們由 config.yaml 中 auxiliary: 區塊獨立配置。

技術細節

Auxiliary task 清單（今日全部修復）：

  • compression — 對話壓縮 summarization
  • vision — 圖片分析（非主模型的 vision call）
  • session_search — 歷史對話搜尋與摘要
  • web_extract — 網頁內容擷取
  • skills_hub — 技能庫搜尋
  • approval — 命令審批輔助判斷
  • mcp — MCP 服務輔助
  • title_generation — 自動生成 session 標題
  • triage_specifier — 對話分類與路由
  • curator — 技能生命週期整理

關鍵原則：auxiliary 唔會自動繼承主模型設定。如果改咗主 provider， auxiliary 仲係指向舊 provider，就會集體失效。

3.3 Persistent Memory vs Session Search

今日用戶問了一個精準的問題：「唔做 session search，你既唯一線索係咩？」 答案是 persistent memory——分為兩塊：

技術細節

MEMORY（個人筆記）

  • 容量上限：2,200 字符（而家 99% 滿）
  • 內容：項目細節、工具怪癖、環境設定、教訓
  • 壽命：長期，每次對話自動 inject

USER PROFILE（用戶檔案）

  • 容量上限：1,375 字符（而家 97% 滿）
  • 內容：偏好、性格、溝通風格、身份
  • 壽命：長期，每次對話自動 inject

SESSION SEARCH（歷史搜尋）

  • 內容：過去每一句對話的完整記錄
  • 壽命：90 日（retention_days 設定）
  • 使用時機：需要「尋日講過咩具體細節」時

簡言之：Memory 記得「你係邊個」，Session search 記得「你尋日講過咩」。

3.4 為何 auto provider 會突然失效？

Hermes 的 provider: auto 有一條內部 fallback chain 和 model lookup table。 當 auxiliary task 設為 auto 時，系統會根據這個 table 自動揀選模型。 問題是：這個 table 可能引用咗已下架、改名、或你帳號無權訪問的 model ID。

技術細節

今次個案：

  • 用戶主模型：kimi-k2.6（Moonshot direct）
  • auxiliary.compression 設為 auto
  • auto 邏輯 resolve 到 kimi-k2-turbo-preview
  • 呢個 model name 不存在於用戶帳號 → 404

原因推測：Moonshot 最近下架或改名了某個 model； 或者 Hermes 的內建 table 更新後引入了 stale reference； 又或者用戶轉 provider 後，auto 的優先次序變咗。

結論：用 Moonshot/Kimi 時，永遠明確指定 auxiliary provider， 唔好依賴 auto。

05Reflection

5.1 關於系統的脆弱性

一個 provider 轉換，影響了 10 個隱藏服務。這是分散式系統的常見問題： 表面上一個設定，底層有無數依賴。用戶從 NVIDIA NIM 轉到 Moonshot direct 時， 只改了主模型，沒意識到 auxiliary 區塊仍然指向舊 provider。

5.2 關於「考考你」

用戶今日的主動測試（「考考你」）證明了一點： 最好的系統維護不是等東西壞了再修，而是主動驗證「如果 X 失效，Y 還能運作嗎？」 這種壓力測試思維，是零代碼用戶最強大的技術直覺。

5.3 關於未來

今日的經驗告訴我們，任何 provider 變更都應該跟隨一個「config 審計清單」： 主模型、fallback、auxiliary、delegation、MCP——逐項確認。 下次轉 provider 時，不再是「改一個地方」，而是「改一張清單」。
