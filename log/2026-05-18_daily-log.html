指令紀錄 Command Reference

連線指令 安全 
ssh root@76.13.216.229
用途：從自己電腦遠端登入 VPS 主機。root 是超級管理員帳號，@ 後面接 IP 位址。首次連線會問是否信任主機，輸入 yes 後再輸入密碼即可。
容器管理 安全 
docker ps
用途：列出「正在運行」的 Docker 容器。顯示容器 ID、使用的映像檔、狀態、端口映射等。若要看到已停止的容器，則用 docker ps -a。
日誌排查 安全 
docker logs filebrowser-filebrowser-1 --tail 20
用途：查看指定容器的日誌輸出。--tail 20 表示只看最後 20 行，避免資訊過量淹沒重點。這是排查問題時最重要的一步——程式不會說謊，日誌會誠實記錄一切。
進入容器 需注意 
docker exec -it filebrowser-filebrowser-1 sh
用途：進入正在運行的容器內部。-i 保持互動，-t 分配虛擬終端。sh 是容器內的 shell。進去後可以像操作普通 Linux 一樣瀏覽檔案系統。執行 exit 即可離開，不會影響容器運行。
重啟服務 需注意 
docker restart filebrowser-filebrowser-1
用途：重新啟動指定容器。這會短暫中斷服務（幾秒鐘），但配置變更通常需要重啟才生效。若容器設定了 restart: always，即使宿主機重開機也會自動啟動。
網路測試 安全 
curl -I http://localhost:8085
用途：只取得 HTTP 回應標頭（Header），不下載內容。-I 是「head」之意，用來快速檢查狀態碼（200/404/403 等）。localhost 代表本機，繞過網路與防火牆，直接測試服務本身是否正常。
內容測試 安全 
curl http://localhost:8085 | head -20
用途：實際取得網頁內容，並用 head -20 只顯示前 20 行。這次排查的關鍵發現：curl -I 返回 404，但 curl（GET）卻成功返回 HTML——說明服務活著，但某些請求路徑有問題。
配置檢查 安全 
cat /config/settings.json
用途：在容器內查看 FileBrowser 的配置檔。重點欄位：baseURL（若設為子路徑會導致根目錄 404）、port（容器內監聽端口）、database（用戶資料庫位置）。
環境變量 安全 
docker inspect filebrowser-filebrowser-1 --format='{{range .Config.Env}}{{.}}{{"\n"}}{{end}}'
用途：查看容器啟動時的環境變量。許多 Docker 映像會用環境變量覆蓋設定檔，例如 FB_BASEURL。若設定檔正確但行為異常，極有可能是環境變量在覆蓋。
檔案檢視 安全 
docker exec filebrowser-filebrowser-1 ls -la /srv
用途：不進入容器，直接執行單一命令並返回結果。這裡檢查 /srv 是 FileBrowser 預設的根目錄（可視為「檔案櫃」），確認裡面有沒有檔案、權限是否正確。

重點詳解 Deep Dive

Featured Command · 01 高頻使用 
docker logs <container_name> --tail <n>
何時使用

服務看似運行卻行為異常、頁面報錯、容器重啟循環、任何「明明開著卻不對勁」的時候。

關鍵參數

--tail N：只看最後 N 行；-f：持續追蹤（像看直播）；--since 10m：只看近 10 分鐘。

排查心法

日誌不會說謊。若日誌說「Listening on port 80」，代表程式層沒問題；若之後出現 404，問題在請求路徑或前端。

常見陷阱

容器名稱打錯、用了 docker logs 但容器根本沒運行（先用 docker ps 確認）、日誌太多淹沒重點（務必加 --tail）。

Featured Command · 02 核心診斷 
curl -I http://localhost:<port>
分層測試思維

網路問題分多層：瀏覽器 → 網路 → 防火牆 → 伺服器 → 服務本身。localhost 繞過前三層，直接測最後一層。

-I vs 普通 curl

-I 只看標頭，快；普通 curl 看內容。本次排查的關鍵：-I 回 404，普通 curl 卻回 HTML——揭露了 HEAD 與 GET 請求行為不一致。

狀態碼速查

200 OK｜301/302 重定向｜400 請求格式錯｜401/403 權限不足｜404 找不到｜500 伺服器內部錯誤｜502/503/504 反向代理出錯

延伸技巧

curl -v 顯示完整交過程；curl -L 跟隨重定向；curl -H "Host: example.com" 模擬域名訪問。

Featured Concept · 03 理解關鍵 
Docker 容器 ≠ 宿主機 ≠ 其他容器
隔離性

每個容器是獨立王國。Hermes 容器見不到 FileBrowser 容器，反之亦然。它們共享宿主機的 Linux 核心，但檔案系統、進程、網路（原則上）彼此隔離。

端口映射

0.0.0.0:8085->80/tcp 的意思是：「宿主機的 8085 號門，通向容器的 80 號門。」外部訪問 8085，Docker 會轉發到容器內的 80。

權限問題

本次排查發現 Hermes 容器內的 hermes 用戶（uid=10000）沒有 docker.sock 存取權，因此無法執行 docker 命令管理其他容器。

為何要 SSH

要管理 Docker，必須「站在宿主機的角度」——即通過 SSH 登入 root，才能看見所有容器、所有設定、所有日誌。

解決邏輯 Problem-Solving Logic

現象觀察：頁面 loading 轉不停
不急着猜原因，先確認「服務到底有沒有在跑」。
確認容器狀態：docker ps
FileBrowser 容器顯示 Up (healthy)，端口映射正確。初步排除「服務死掉了」。
結論：服務活着，問題在「為什麼活着卻不回應」。
讀取日誌：docker logs
日誌顯示 Listening on port 80，但歷史記錄有 /: 404。這是一個矛盾：服務說它在聽，但訪問根路徑卻找不到。
結論：程式層正常，但請求處理有異常。
分層測試：curl localhost
在宿主機執行 curl，繞過網路與防火牆，直接測試服務本身。發現 GET 請求返回完整 HTML，但 HEAD 請求返回 404。
結論：服務確實能回應，但前端載入後可能因某些 API 失敗而卡住。
誤入歧途：懷疑 baseURL 設定錯誤
由於日誌有 404，第一時間懷疑是 FileBrowser 的 baseURL 設定問題（這是常見病因）。進入容器檢查 settings.json，發現 baseURL 是空字串，完全正確。
假設錯誤。排除 baseURL 問題，繼續追查。
深入測試 API 端點
測試 /api/resources 返回 403 Forbidden，/api/login 正常。這說明 FileBrowser 需要登入才能使用 API，但前端應該自動跳轉登入頁。
結論：服務邏輯正確（要登入），但前端載入流程卡住了。
突破：直接訪問 /login
既然首頁卡住，嘗試繞過首頁直接訪問登入路徑。結果成功載入登入頁面。
最終診斷：瀏覽器快取或前端路由處理問題，導致首頁 JavaScript 執行異常，卡在 loading。直接訪問 /login 即可繞過。

附帶收穫 Collateral Learning

SSH 連線的現實

防火牆規則改完後，有時需要等待數分鐘才會生效。第一次 timeout 不要慌，確認規則正確後，等一陣再試。

瀏覽器快取的欺騙性

開發者工具的 Network tab 全部顯示 200，不代表真的沒問題——可能是快取騙了你。永遠用「無痕模式」或「Ctrl + Shift + R」作對照測試。

日誌裡的 404 不一定是錯

FileBrowser 的 /api/resources 對未登入者回 403，這是正確行為。要學會分辨「錯誤的 404」和「預期中的拒絕」。

Hermes 瀏覽器是「用完即棄」

Agent 的 browser_navigate 預設使用臨時 Chromium session，不會永久保存 cookies。但 /tmp 裡會遺留通訊目錄殘骸，定期重啟容器即可清理。
